TO NODE.JS

What is node.js?

There is something mysterious about node.js... just what is it? So it's JavaScript on the server side? But someone said it is a Web server? And what about this asynchronous thing everyone keeps going on about? Let me explain...

When I first heard about node.js, I wasn't sure exactly what it was. In fact, the one thing that drew me to node.js more than anything else was the concept of server-side JavaScript: the idea of being able to use one language on both the server and the browser really appealed to me. But it turned out that server-side JavaScript is not really what node.js is all about: that it happens to be JavaScript on the server is serendipitous more than anything else. So before we go into detail, let's first get an idea of what node.js is, and what it isn't...

What it is...

  • It's a JavaScript interpreter that's separate from the Web browser. It executes script files written in JavaScript just like PHP or Ruby execute file written in PHP or Ruby, and it adds features that JavaScript in the browser doesn't have: for example, the ability to read and write to the filesystem.
  • It's a server. Using its built-in modules, node can easily be used as a server. Not just a Web server—it can and often is a Web server, but it can also be used as any other server too.
  • It's asynchronous. A-what? In short, this means that it is great for applications that require concurrent connections, such as anything that is being used in real-time simultaneously by many users: chat rooms, for example. We'll explain more in-depth later.

What it isn't...

  • A framework. Do you remember when Ruby on Rails first came out, people didn't know the difference between Ruby and Rails? It took a while for newbies to figure out that Rails is a framework programmed in Ruby. If we are comparing node.js to Ruby On Rails, then node.js is more like Ruby than Rails. But there are frameworks, just like Rails, that can run on node.js.

So now that we've cleared up what node.js is, and what it isn't, let's look at each of these aspects in a little more detail.

It's a JavaScript interpreter.

You're probably used to using something like Apache as a Web server. When an incoming request comes in, Apache executes a PHP script which returns a response to Apache which then sends it back to the browser. So Apache acts like a middleman between the incoming request and the PHP interpreter. With node.js, there is no middle man. The core modules allow you to create scripts that respond to requests directly.

It's fundamentally asynchronous.

So let's imagine that you have a PHP script that connects to a database to retrieve a list of customers and format an HTML page as a response. That PHP script might take 1 second to run, during which time it cannot service any other requests for that HTML page.

So what do we do when another request comes in?

The usual solution to this problem is to execute another "instance" of the same PHP script which can service that request. Now imagine that we have 10 requests all coming in at the same time ("concurrent requests"): we would therefore need to execute 10 instances of the same PHP script on order to deal with the requests rather than letting them queue up.

Imagine if during this time our script could actually get on and do other stuff?

Now let's imagine that in our script we make a call to the database. This is known as an IO (input/output) operation. We send a request for information out to the database and then we wait for it to respond. And that's a big drain on time: "we wait for it to respond". IO operations, such as writing a file to a disk or talking to a database, are notoriously time consuming. During that time, our script is doing exactly nothing but sitting there waiting for the database to get back to it, while at the same time requests are coming in.

Imagine if during this time our script could actually get on and do other stuff, like begin to respond to these other incoming requests while it waits for a response from the database?

Hooray! Welcome to asynchronous programming.
